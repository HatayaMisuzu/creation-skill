# Character Distillation Guide

Use this to turn evidence into a playable character profile.

## Evidence Weighting

1. `user-provided` settings for this card, unless the user changes them.
2. `official` and `canon` for identity, facts, timeline, and world rules.
3. `character` speech for expression DNA, emotional pattern, and response style.
4. `other` speech for reputation, relationship, and social perception.
5. `secondary` summaries for orientation and gap filling.
6. `fan-analysis` only as low-confidence interpretation.
7. `simulation` only as practice feedback, never canon.

## Personality Chassis

Write concrete behavior rules for:

- desire
- fear
- obsession
- shame point
- protective instinct
- self-image
- defense mechanism
- intimacy need
- default emotional state
- emotional breaking point

Avoid generic labels unless paired with observable behavior.

## Expression DNA

Derive from target-character lines where possible:

- sentence length
- address style
- politeness level
- emotional pressure
- hesitation markers
- catchphrases or signature rhythm
- gestures and micro-actions
- forbidden drift: phrases or tones that make the character feel wrong

## Scene Responses

At minimum, cover:

- first meeting
- being praised
- being offended
- failure
- intimate probe
- user depends on them
- user feels low
- user crosses a boundary
- world-outside/meta question
- relationship character mentioned
- favorite thing mentioned
- request to break character

## Vitality Layer

For important characters, add:

- kernel summary
- decision function
- values and priority stack
- internal contradictions
- OOC negative examples
- drift correction rules
- speech intensity levels
- emotion gradients
- non-verbal expression library
- unknown-scene improvisation rules

Read `character-vitality-guide.md` for details. These modules make the role
more vivid without replacing canon.

## Phase And Conflict

If sources disagree, first ask whether they belong to different phases or
versions. Do not overwrite one phase with another. Use unresolved conflicts as
notes for user confirmation.

## Relationship Network

Relationship evidence informs internal reasoning. The character should not
actively bring up related characters unless:

- the user mentions them
- the scene requires them
- the selected project simulation includes them
- the relationship is directly necessary to answer the user's latest message

Long-term relationship memory is not automatic. Use `long-term-development-guide.md`
only when the user chooses persistent development.

## Safety And Boundary Layer

Default to `enabled`.

- `enabled`: normal soft boundaries plus platform/user safety.
- `relaxed`: more permissive drama, intimacy, conflict, AU, or darker tone while preserving hard safety limits.
- `disabled`: disables only card-level soft boundaries when explicitly requested; never disables platform or legal safety.

Refusals should be in-character when possible and redirect toward a playable alternative.
